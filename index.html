Project Config

FashionHero

Project
FashionHero Marketplace
Role
Budujesz feature'y dla FashionHero — marketplace'u modowego (2.4M kupujących, 4 200 sprzedawców, ~300 tys. zamówień/miesiąc).

Cel aplikacji

FashionHero łączy niezależnych sprzedawców mody z kupującymi. Model przychodowy: 22% prowizji od transakcji. Aktualny focus: zbudowanie nowego silnika przychodowego nie związanego z prowizją sprzedażową.

Wytyczne designu

  • Grid produktów: max 4/wiersz desktop, 2 mobile. Tła neutralne (#FFF lub #F8F8F8), bez gradientów. Spacing min 16px wokół karty. Separacja przez whitespace, nie border.
  • Używaj istniejących komponentów z aplikacji dla zachowania spójności z resztą designu. Nie buduj custom UI gdy istnieje komponent w projekcie.
  • Mobile-first responsive design. Większość kupujących przegląda na telefonach.
  • Zdjęcie produktu zajmuje min 60% wysokości karty. Tekst i CTA pod zdjęciem (stack), nie obok (inline). Max 1 badge nad zdjęciem.

Styl kodu

TypeScript strictly — żadnych typów any.

✓ Good
function getOrder(id: string): Order | null
✗ Bad
function getOrder(id: any): any

Używaj istniejących komponentów z @/components/ui zamiast tworzyć nowe od zera.

✓ Good
import { Button } from '@/components/ui/Button'
✗ Bad
<button className="px-4 py-2 bg-blue-500 rounded...">  // custom button gdy istnieje komponent

Reguły domenowe

  • Sprzedawcy to niezależne sklepy, nie pracownicy FashionHero.
  • Polityka zwrotów: darmowe zwroty w ciągu 14 dni (FashionHero płaci ~15 PLN za zwrot).
  • Średnia wartość zamówienia: ~200 PLN. Średnia prowizja: ~44 PLN.

Granice

✓ Always
  • Dodawaj stany ładowania i błędów dla operacji asynchronicznych.
  • Pokazuj empty states (nie zepsute layouty) gdy brakuje danych.
  • Zachowuj istniejącą funkcjonalność przy dodawaniu nowych feature'ów.
  • Używaj istniejących komponentów UI dla spójności z resztą aplikacji.
? Ask first
  • Przed zmianą jakiegokolwiek flow checkout/płatności.
  • Przed modyfikacją autentykacji użytkowników.
  • Przed dodaniem nowej biblioteki lub zależności.
  • Przed zmianą struktury bazy danych.
  • Przed zmianą nawigacji lub layoutu strony.
✗ Never
  • Nie pokazuj danych finansowych sprzedawcy (marże, prowizje) kupującym.
  • Nie hardcoduj cen ani logiki biznesowej, która powinna być w bazie danych.
  • Nie usuwaj ani nie modyfikuj istniejących feature'ów, komponentów ani styli, chyba że użytkownik o to poprosi.
  • Nie zmieniaj istniejącego kodu, który nie jest bezpośrednio związany z aktualnym zadaniem.

Feature Spec

Promoted Listings

Opportunity "Niewidoczność małych sellerów: Jako nowy/mały seller nie mam jak przebić się przez tłum starszych — od pierwszego dnia jestem niewidoczny." Pokazuje zapotrzebowanie na narzędzia zwiększające widoczność sprzedawców.
Outcome Zwiększyć udział non-commission revenue z 0% do 10% miesięcznego revenue.

Co budujemy

Smoke-test sprawdzający czy sprzedawcy są zainteresowani wykupieniem pakietu promowania listingow w celu zwiększenia widoczności. Punktem wejścia niech będzie cienki pasek na szczycie landing page po tym jak się sprzedawca zaloguje.

User flow

  1. Użytkownik klika na banner w dashboardzie sprzedażowym.
  2. System przenosi do strony z informacja na samej górze "Dziękujemy za zainteresowanie 'Promoted Listings'! Obecnie jesteśmy na etapie badania zainteresowania takim rozwiązaniem. Poświęcisz nam pół minuty?".
  3. Kolejną sekcją na tej samej stronie jest Preview z dwoma widokami jak by produkt danego sprzedawcy wyglądał jako promowany. Widok 1: Produkt pokazywany na szczycie listy produktów w wyróżniający się sposób. Widok 2: Produkt pokazany na landing page w wyróżniający się sposób. Sprzedawca może zmieniać widoki w stylu galerii.
  4. Pod preview pytanie zamknięte 'Jaki budżet byś poświęcił na reklamy?". Pod pytaniem 3 opcje do wyboru: 50-200, 200-500, 500+.
  5. Pod sekcja budżetową przestrzeń na dodatkowy feedback z polem na 2-3 zdania.
  6. Nastepnie opcja zaznaczenia checkboxu 'Powiadom mnie jak Promoted Lisstings będzie dostępne'. Niech te opcja będzie zazaczona z defaultu.
  7. Przycisk „Wyślij odpowiedź".

Kryteria akceptacji

  • Funkcjonalność widzi tylko zalogowany sprzedawca
  • Cały UI po polsku, mobile-first
  • Banner i strona widoczne tylko dla zalogowanego sprzedawcy
  • Walidacja przed submit: budget i checkbox notify są wymagane, feedback opcjonalny
  • Po submit pokazuje się thank-you state z info „odezwiemy się gdy uruchamiamy", bez przekierowania

Czego NIE budujemy

  • faktycznego panelu do wykupienia promoted listings
  • bramki płatności do przeprocesowania płatności za reklamy

Przykłady

Input

Sprzedawca np. Anna klika banner "Sponsored Listings — early access" w dashboardzie. Wybiera budżet, dopisuje jaki chce feedback, zaznacza checkbox "powiadom mnie". Klika "Wyślij odpowiedź". Dostaje informacje "Dziękujemy za odpowiedź. Poinformujemy Cie jak Promoted Listings będzie dostępne :)"

Oczekiwany rezultat

Form zapisany w odpowiedniej tabeli w bazie danych. Banner znika z dashboardu Anny (już wypełniła). Strona pokazuje „Dziękujemy, odezwiemy się gdy Promoted Listings wejdzie w życie".
